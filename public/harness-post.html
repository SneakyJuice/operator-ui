What Is a Harness?

A harness is every piece of code, configuration, and execution logic that isn't the model itself. A raw model is not an agent. It becomes one once a harness gives it state, tool execution, feedback loops, and enforceable constraints.

📋
System Prompts & Memory Files
AGENTS.md, skill files, subagent prompts — the operating rules the model follows on every session start.
🛠️
Tools, Skills & MCP Servers
The actions the agent can take — bash execution, API calls, browser, file operations, third-party integrations.
🔗
Orchestration Logic
Subagent spawning, model routing (right model for right task), handoffs, parallel execution, task decomposition.
🔐
Hooks & Enforcement Layer
Scripts that run at lifecycle points — block destructive commands, require approval, run tests after every edit, enforce compliance.
📡
Observability
Logs, traces, cost metering, latency tracking — visibility into what the agent actually did, not just what it said.

The 4 Pillars

💉
Context Injection
Prompts, memory, skills, conversation history — delivered at query time so the model always works with ground truth, not stale weights.
🎯
Control
Compaction, orchestration, feedback loops. Managing what the agent does, when it stops, and how it recovers from failure.
🔍
Observe & Verify
Screenshots, tests, logs — closing the self-verification loop so the agent can check its own output before calling done.
💾
Persist
Filesystem, git, progress files — durable state across sessions. No amnesia. Every session picks up where the last left off.

The Skill Issue Reframe

The harness-engineering mindset rejects "blame the model." Most agent failures are legible: the agent didn't know a convention → add it to AGENTS.md. The agent ran a destructive command → add a hook that blocks it. The agent kept "finishing" broken code → wire a typecheck back-pressure signal into the loop.

On Terminal Bench 2.0, Claude Opus 4.6 running in Claude Code scores far lower than the same model in a custom harness. One team moved from Top 30 to Top 5 by changing only the harness.

The Ratchet Principle

Every agent mistake is a permanent signal, not a story to laugh about. When the agent fails → engineer a fix → it never makes that mistake again. Every line in a good AGENTS.md is traceable to a specific failure. You can't download the right harness. It's shaped by your failure history.

Where This Is Going: HaaS

Harness-as-a-Service (HaaS)
We're moving from LLM APIs (which give you a completion) to harness APIs (which give you a runtime). The Claude Agent SDK, Codex SDK, and OpenAI Agents SDK all point in the same direction — you get the loop, the tools, the context management, the hooks, and the sandbox primitives out of the box. Your job is to configure and customize them for your domain.

The shift: Stop debating which model to use. Start engineering the system around it.
The model is one chip on the board.
The harness is everything else that makes it useful.
Harness engineering is a discipline — not a framework you download. The teams winning in 2026 aren't the ones with the best model access. They're the ones with the tightest harness, the sharpest feedback loops, and the discipline to treat every agent failure as an engineering input.
